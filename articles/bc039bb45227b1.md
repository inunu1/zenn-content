# 1. はじめに（Introduction）

逐次計算モデルは、時系列データを前から順に処理することで状態を更新する手法であり、  
Elo レーティングはその典型例である。  
しかし、実際のアプリケーションでは過去のデータが後から修正・追加されることが多く、  
逐次更新では整合性が保てないという課題がある。

この問題は、レーティングのように「過去 → 現在 → 未来」が連鎖するモデルにおいて特に顕著である。  
途中のデータが変わると、その後のすべての計算結果が変化するため、  
部分的な更新では正しい状態を維持できない。

本記事では、この逐次性を前提としたうえで、  
**フル再計算を高速に行い、差分のみをバルク UPDATE する手法**を提示する。  
これにより、整合性を保ちながら高速に再計算できる仕組みを構築する。

本記事の貢献は以下の通りである。

1. Elo レーティングの逐次性がバッチ処理に与える影響を整理した。  
2. 全件再計算を O(N) で実現するメモリ逐次更新手法を提示した。  
3. データ取得の一元化と生SQLバルク更新により、DB I/Oの回数と負荷を最小化した。  
4. 実測により、約 300 件の対局データを 40ms でフル再計算可能であることを示した。

---

# 2. 背景（Background）

## 2.1 逐次計算モデル（Online Sequential Model）

逐次計算モデルは、時系列データを前から順に処理し、直前の状態を用いて次の状態を更新する方式である。  
このモデルは「過去 → 現在 → 未来」と一方向に依存するため、途中のデータが変わるとその後のすべての計算結果が変化するという特徴を持つ。

Elo レーティングはこの逐次性を強く持つ典型的なアルゴリズムである。

- 新しいデータが来るたびに更新  
- 過去の状態に依存  
- 一部のデータ変更が全体に波及  

この構造は理論的には正しいが、実務上は「部分更新ができない」という制約を生む。

---

## 2.2 ORM 逐次 UPDATE の課題

一般的な ORM（Object Relational Mapping）による逐次 UPDATE 実装では、  
各対局ごとにレートを計算し、結果を即座に DB に反映する。  
しかしこの方式では、対局数 N に対して **2N 回の I/O** が発生し、  
データ量が増えるほど処理時間が線形に増加する。

| 処理方式 | I/O 回数 | 特徴 |
|-----------|-----------|------|
| ORM 逐次 UPDATE | 2N | 各対局ごとに即時更新、整合性は保たれるが遅い |
| フル再計算 + 差分 UPDATE | 2 | メモリ上で全件再計算後、差分のみ更新 |

ORM 方式は「正確だが遅い」。  
特にクラウド環境（Neon など）ではネットワーク遅延が支配的となり、  
1280 件のデータで約 4.8 秒を要することが確認された。

---

## 2.3 スナップショット方式の限界

一部のシステムでは「スナップショット方式」によって  
途中の状態をキャッシュし、再計算を部分的に省略する設計を採用している。  
しかし Elo のような逐次モデルでは、  
過去の 1 件でも変更が入ると全体の再計算が必要になるため、  
スナップショットは整合性を保証できない。

---

## 2.4 全件再計算の必要性

逐次モデルにおいて整合性を保つ唯一の方法は、  
**全件を再計算すること**である。  
ただし、これを単純に実装すると処理時間が膨大になるため、  
本記事では「メモリ上でフル再計算 → 差分のみ UPDATE」という  
最適化手法を提案する。

---

# 3. 提案手法（Proposed Method）

本記事で提案する手法は、逐次計算モデルの本質である「全件再計算の必要性」を受け入れたうえで、計算を完全にインメモリ化し、DB I/Oを極限まで削減するアプローチである。

## 3.1 処理フロー

全体の処理フローは以下のステップで実行される。データ取得フェーズにおけるDBとの二重フェッチ（重複取得）を徹底的に排除し、通信回数を**計2回（定数回）**のみに抑え込んでいる。

1. **データ一元フェッチ**: DBアクセスを最小化するため、最初から時系列順（`matchDate ASC, roundIndex ASC`）に整列された全対局データ、および全プレイヤーデータを一度に1回だけロードする。
2. **状態のマッピング**: 取得した既存データを比較用のインメモリMap構造に展開し、各プレイヤーの計算用ワークエリアを初期マスタレート（`initialRate`）で初期化する。
3. **インメモリ逐次計算**: ループ処理内で、対局直前のレート（スナップショット）を配列に記録しつつ、Eloアルゴリズムを適用して次局用の現在レートを順次計算・更新していく。
4. **論理差分抽出（Diffing）**: 計算完了後の最新データと、事前に取得していた既存のDBデータをメモリ上で全件比較し、実際に値が変化したレコードのみを抽出する。
5. **高速バルクUPDATE**: 抽出された差分データをSQLの `VALUES` 句を用いた生SQLバルククエリにまとめ、1回のクエリでDBへ一括反映する。

---

## 3.2 差分抽出とバルク UPDATE

インメモリで計算された状態をそのままDBに上書きすると、変更のないレコードに対しても無駄な書き込みが発生し、DBの負荷となる。そのため、差分抽出により実際に値が変化したレコードのみを更新対象に指定する。

抽出された差分データは、ORMによる逐次的な `update` 文の発行ではなく、1発の強力な生SQLバルククエリへと結合され、アトミックにDBへ反映される。

これにより、対局数 $N$ に対するDB I/Oの回数は一気に **$2$ 回（Resultテーブルで1回、Playerテーブルで1回）**へと削減され、書き込み効率は $1 - \frac{1}{N}$ 削減（$N=1000$ の場合、**99.9%削減**）される。

---

# 4. 評価（Evaluation）

提案手法の有効性を検証するため、Next.js（Vercel）とサーバーレスPostgreSQL（Neon）の環境を用いて本番環境での実測評価を行った。検証には、プレイヤー64名、対局データ294件のデータセットを使用した。

## 4.1 実測結果（Performance Measurement）

過去のタイムラインに対して、連鎖的なレート変更が発生するデータ更新を行った際の、各フェーズの詳細なプロファイリング結果は以下の通りである。

### ✔ フル再計算および論理差分一括更新処理の処理時間（全294件）
| 処理フェーズ（プロファイルラベル） | 実行時間 (ms) | 業務処理内容・備考 |
| :--- | :--- | :--- |
| `phase1_fetch_db_data` | 19.5 ms | DBからのデータ一元フェッチ・Map構築 |
| `phase2_elo_in_memory_loop` | **0.25 ms** | インメモリでのElo逐次計算ループ（全件） |
| `phase3_extract_logical_diffs` | 0.05 ms | メモリ上での新旧データ論理差分抽出 |
| `phase4_db_bulk_update` | 19.9 ms | 差分（対局: 116件、プレイヤー: 17名）の一括反映 |
| **総処理時間（`total_execution_time`）** | **40.0 ms** | **目標の50ms未満を余裕でクリア** |

実測データが示す通り、294件の過去データに対してわずか1件の変更を加えた結果、ドミノ倒しのように**全体の約40%にあたる116件の対局のレートスナップショットが書き換わった**。それだけの激しい連鎖が発生してもなお、全体の処理は40.0msという極めて高速な時間で終了している。

---

## 4.2 スケール特性（Scaling Behavior）

実測データから、メモリ上の計算コストは極めて小さく、処理時間の大部分をDB I/Oが占めていることが分かる。全体の処理時間 $T(N)$ は、データ量 $N$ に対して以下の線形モデルで近似できる。

$$T(N) \approx aN + b$$

ここで、$a$ は逐次計算・差分抽出の係数（約 $0.001 \text{ ms/件}$）、$b$ はDB I/Oの固定コスト（約 $30 \sim 40\text{ ms}$）である。このモデルに基づくと、データ量が増加した場合の予測時間は以下のようになる。

- **1,000 件:** 約 41 ms
- **10,000 件:** 約 50 ms
- **100,000 件:** 約 140 ms
- **1,000,000 件 (100万件):** 約 1,040 ms（約1秒）

メモリ上の計算自体は100万件でも1秒前後で終了するため、破綻することなく線形スケール（$O(N)$）する特性を持つ。

---

## 4.3 トランザクション不要設計の妥当性

本提案手法では、DB更新時に明示的な一連のトランザクション（`BEGIN / COMMIT`）を張っていない。保存前後の整合性に懸念を持つかもしれないが、これは以下の理由から実務上、妥当かつ合理的である。

1. **メモリ完結型:** すべての計算がメモリ上で完了してからDBに書き込むため、DB側に「計算途中の不整合な状態」が書き込まれるリスクがそもそも存在しない。
2. **依存関係の排除:** 差分抽出後のResultテーブルとPlayerテーブルの更新は、データ構造的に同時性を厳密に保証する必要がない。
3. **低コストな再試行:** 万が一書き込み中にエラーが発生した場合でも、処理全体が40ms前後と極めて高速なため、バッチを最初から再実行（リトライ）するだけで安全に最新状態へと復旧できる。

# 5. 考察（Discussion）

## 5.1 部分更新できない逐次モデルの宿命

Eloレーティングにおいて、過去のわずか1件の変更に対して、差分結果が116件も発生したという実測事象は、このモデルの「宿命」を明確に示している。

過去のたった1つの星勘定（勝敗）が変わるだけで、その後に紐づくプレイヤーたちの初期レートが変わり、対局ごとの期待勝率が変わり、最終的な現在レートに至るまでがドミノ倒しのように連鎖して変化する。この強固な依存関係がある以上、**「変更があった周辺だけを部分更新する」というアプローチは数学的に不可能**であり、全件再計算こそが唯一 of 唯一の正攻法となる。

---

## 5.2 フル再計算は「重い」という誤解

エンジニアは「全件再計算（フルリビルド）」と聞くと、直感的に「重い処理」「避けるべきアンチパターン」と捉えがちである。しかし、今回の実験が証明した通り、約300件のデータに対する純粋な「計算コスト」は **0.25ms** であり、全体の処理時間の1%にも満たない。

重さの本質は「計算」ではなく、ORMが裏側で発行する膨大な「DB I/O」であった。アルゴリズムが $O(N)$ であっても、CPUの演算能力に対してデータサイズが十分に小さければ、インメモリ処理は一瞬で終わる。「全件再計算 ＝ 重い」という先入観は、I/Oボトルネックと計算ボトルネックを混同した誤解であると言える。

---

## 5.3 I/O 削減設計の本質

本手法のブレイクスルーは、逐次計算モデルの「部分更新できない」という数学的制約を認めながらも、**「DBへのアクセスと書き込みは定数回（$O(1)$）に抑え込める」**というエンジニアリング的な最適化を両立させた点にある。

データの一元フェッチによってフェッチ通信を1回にまとめ、さらにどれだけ過去のデータが激しく修正されようとも、バッチ処理中にDBをロックする時間は生SQLバルクUPDATEの一瞬のみである。これにより、データの整合性を100%担保しながら、実用的なパフォーマンスを担保する基盤が実現した。

---

# 6. まとめ（Conclusion）

## 6.1 成果の整理

本記事では、過去データの遡及修正が頻繁に発生する実務の課題に対し、Eloレーティングを題材にした高速な逐次バッチ更新手法を提示した。

- 逐次モデルの整合性を保証するため、あえて**「全件再計算」**を採用。
- 処理を一元フェッチとインメモリに閉じ込めることで、計算時間を **0.25ms** に抑制。
- 新旧データの差分抽出（Diff）とバルクUPDATEにより、DB通信を極限まで最適化。
- 結果として、過去のデータ変更に対しても**整合性を完全に保ちながら、40msという超高速な再計算**を達成した。

---

## 6.2 他モデルへの応用可能性（Glicko / TrueSkill）

本手法のコア思想である「メモリ上での時系列フル再計算 ＋ 差分バルクUPDATE」という構造は、Eloレーティング固有のものではない。

「過去の状態が未来の状態を決定する」という一方向の逐次性を持つレーティングモデル、例えば**Glicko（グリコ）**や、その発展形である**Glicko-2**、さらには多人数・多チームに対応した**TrueSkill（トゥルースキル）**など、あらゆるレーティングアルゴリズムにそのまま適用可能である。特にTrueSkillは、確率伝播（Factor Graph）を用いるためEloよりも計算負荷が高いが、インメモリで処理を完結させる本手法であれば、実務上全く問題のない速度でバッチ化できる。

---

## 6.3 今後の展望

本手法は非常に強力であるが、データ量が数百万、数千万件規模に達した場合には、インメモリにデータを乗せきるための「メモリ消費量」および「DBからのデータ取得（Fetch）のネットワーク遅延」が次のボトルネックになることが予想される。

今後の展望としては、全件を取得するのではなく、**「過去の変更が発生した最古の時点」を特定し、それ以降のタイムラインのみを限定して取得・再計算する「部分リビルド（区間全件再計算）」**への拡張が挙げられる。これにより、逐次モデルの正しさを維持したまま、データ量が数億件にスケールしても数ミリ秒で処理を終える、さらなる堅牢なバッチ基盤の構築が可能になるだろう。
