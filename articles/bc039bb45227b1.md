# Eloレーティング再計算バッチを高速化する ― I/Oボトルネックの解消とアーキテクチャ設計

## 1. はじめに
Eloレーティングに代表される「過去の積み重ねで現在が決まる」逐次計算モデルは、多くのシステムで採用されています。しかし、データ修正時の整合性維持（再計算）は、データ量が増加するにつれて「重いバッチ処理」へと変貌しがちです。

本記事では、単純な「計算の高速化」ではなく、システム全体のI/Oコストを分析し、計測データに基づいた予測モデルから最適なアーキテクチャ設計を解説します。

## 2. ボトルネックの真実：DBアクセスという「見えないコスト」
Eloレーティングの性質上、ある対局結果の修正は、それ以降の全プレイヤーのレートに連鎖的に波及します。多くの実装では、再計算バッチにおいて「読み込み→計算→更新」をループ内で実行してしまいます。対局数を $N$ とすると、このアプローチの計算量は以下のようになります。

- **改善前 (逐次I/O)**: $O(N \times \text{DB Latency})$

ORMを介した逐次更新は、計算ロジック自体のコストよりも「DBとの往復回数」が圧倒的な支配要因となります。プロファイルの結果、真のボトルネックはロジックではなく、ORM経由の不要なI/Oにあることが判明しました。

## 3. インメモリ計算による $O(N)$ への回帰
この問題を解決するため、I/Oを「取得」と「更新」の2回（定数時間）に抑える戦略へ移行しました。

1. **データ一括取得**: 必要な全データを1回のクエリで確保。
2. **インメモリ処理**: Map構造を用いてメモリ上で計算を完結。
3. **論理差分抽出**: 計算結果と既存データを比較し、更新が必要なレコードのみを特定。
4. **バルク更新**: 抽出した差分を1回のクエリで一括永続化。

## 4. パフォーマンス実測と構成比の分析
サーバレス環境（Vercel / Next.js）において、約295件の対局データを用いて10回の計測を行いました。

### 安定稼働時のフェーズ別コスト（スパイク除外平均）
サーバレス特有のコールドスタート等の「揺らぎ」を除いた、純粋なロジック性能の構成比は以下の通りです。

| フェーズ | 平均所要時間 | 全体に占める割合 |
| :--- | :--- | :--- |
| **Phase 1: DBデータ取得** | 25.53 ms | **64.9%** |
| **Phase 2: Eloインメモリ計算** | 0.35 ms | **0.9%** |
| **Phase 3: 論理差分抽出** | 0.23 ms | **0.6%** |
| **Phase 4: DBバルク更新** | 33.22 ms | **33.6%** |
| **合計** | **39.33 ms** | **100%** |

### 考察：1.5% vs 98.5%
注目すべきは、純粋な計算ロジック（Phase 2+3）が占める時間は**全体のわずか1.5%**に過ぎないという点です。残りの**98.5%はすべてDB I/O**です。このデータは、アルゴリズムの最適化よりも、I/O回数を定数回 $O(1)$ に抑え込むアーキテクチャ設計がいかに重要であるかを雄弁に物語っています。

## 5. 線形予測モデルによるスケーラビリティの検証
実測値（39.33ms / 295件）に基づき、1件あたりのコスト $\alpha \approx 0.133 \text{ms}$ とした線形予測モデル $T(N) = \alpha N$ で、将来的なデータ増加を予測します。

| データ件数 ($N$) | 予測処理時間 | 備考 |
| :--- | :--- | :--- |
| **1,000 局** | 約 0.13 秒 | |
| **10,000 局** | **約 1.33 秒** | サーバレスの起動時間に匹敵 |
| **100,000 局** | 約 13.3 秒 | タイムアウト設定の考慮が必要 |

このモデルから、1万局規模であっても処理時間は1秒強であり、サーバレスのコールドスタート時間（今回の計測では最大1.4秒を記録）と同程度であることがわかります。

## 6. アーキテクチャの「割り切り」と戦略
### サーバレス特有のスパイク
計測中、時折1,400msを超える実行時間が確認されました。これはサーバレス環境におけるDB接続の初期化等によるものです。ロジックが30ms台で動いている以上、この1秒強の「揺らぎ」はインフラの特性として許容し、コード側の複雑な最適化を避けるという判断が、保守性の観点から正解となります。

### チェックポイント設計
今後、データが数十万件を超えた場合、バルク更新といえどもI/O限界や関数の実行時間制限に抵触します。その際、本手法と組み合わせて「シーズン制（特定時点のレートをチェックポイントとして保存）」を導入することで、再計算の物理的な範囲を限定し、スケーラビリティを担保する計画です。

## 7. おわりに
計算処理の最適化において重要なのは、アルゴリズムの複雑性だけでなく、「システムが物理リソース（DB I/O）にどうアクセスしているか」を定量的に可視化することです。98.5%のボトルネックがどこにあるかを知ることで、真に打つべき一手が見えてきます。
