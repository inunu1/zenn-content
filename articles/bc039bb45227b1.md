# はじめに
レーティング計算は本来「逐次計算（オンライン更新）」として設計されています。しかし実際のアプリケーションでは、過去の対局データが後から修正されたり、順序が変わったりすることがあります。このような状況では、逐次更新では整合性が保てず、**全件再計算（バッチ処理）**が必要になります。

本記事では、Elo レーティングを題材に、逐次計算モデルをバッチ処理で正しく・高速に扱うための設計手法を解説します。

---

# 1. 逐次計算とは何か
逐次計算（オンライン更新）は、時系列データを前から順に処理し、直前の状態を使って次の状態を更新する計算モデルです。

- 新しいデータが来るたびに更新
- 過去の状態に依存
- 状態遷移モデル（state transition model）

Elo レーティングもこの構造を持ちます。

---

# 2. Elo レーティングの数学的背景
Elo の勝率モデルはロジスティック関数で表されます。



\[
P(A\text{ wins}) = \frac{1}{1 + 10^{-(R_A - R_B)/400}}
\]



更新式は以下の通りです。



\[
R_A' = R_A + K(S_A - E_A)
\]



このように、**未来のレートは必ず過去のレートに依存**します。

---

# 3. なぜバッチ再計算が必要なのか
実際のアプリケーションでは、以下のような「過去が変わる」事象が頻繁に発生します。

- 過去の対局が修正される
- 削除される
- ラウンド順が変わる
- 過去の日付に新しい対局が追加される

これらはすべて **未来のレートに連鎖的に影響**します。

そのため、差分更新では整合性が保てず、**全件再計算が最も安全**になります。

---

# 4. 全件再計算のボトルネック
- 対局数 N に対して O(N)
- プレイヤー数 M に対して O(M) のメモリ
- ORM の逐次クエリは遅い
- ソートや I/O が重い

---

# 5. 高速化のための設計戦略
## 5.1 データを一括取得し、メモリで逐次更新
- 全対局を日付順にソート
- プレイヤーのレートを Map で保持
- 1 パスで更新

## 5.2 ORM を避けて SQL で高速化
- Prisma.sql を使う
- インデックス設計（userId, matchDate, roundIndex）

## 5.3 スナップショットの活用
- 計算中はメモリに保持
- 最後にまとめて upsert
